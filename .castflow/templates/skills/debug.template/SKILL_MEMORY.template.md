## Part 1：硬性规则

### 规则 1：检查深度必须匹配 L1 参数

定义
strictness 参数决定检查范围。Deep 模式必须检查全部清单项，Quick 模式仅检查数据边界的 P0 项。禁止在 Quick 模式下跳过所有检查，也禁止在 Deep 模式下只做部分检查。

检查清单
- [ ] 是否按 strictness 参数选择了正确的检查范围？
- [ ] Deep 模式是否覆盖了全部三个 Category？

### 规则 2：修复建议必须保持功能不变

定义
当 suggest_fixes 为 true 时，生成的修复代码不得改变原有功能逻辑，只能添加防御性检查。

检查清单
- [ ] 修复代码是否只添加了检查，没有改变逻辑？
- [ ] 修复后的代码是否能编译？

### 规则 3：报告必须包含定位信息

定义
每个发现的问题必须包含文件路径、行号（如可确定）、问题分类和严重程度。

检查清单
- [ ] 每个问题是否有文件和位置信息？
- [ ] 是否标注了严重程度（P0/P1/P2）？

## Part 2：常见陷阱

### 陷阱 1：只检查表面不追踪调用链

现象
检查一个方法时只看方法内部的 null 检查，不追踪其调用者是否可能传入 null。

防护
对 P0 级检查项，必须追踪至少一层调用链。

### 陷阱 2：误报过多导致报告无效

现象
Deep 模式下报告了大量低风险问题，淹没了真正的高风险项。

防护
报告按严重程度分组（P0/P1/P2），P0 必须排在最前。
