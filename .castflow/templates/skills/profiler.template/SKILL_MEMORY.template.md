# profiler-skill-memory - 性能检测硬性规则

**本文档的性质**：硬性约束（必须遵守）。

---

## Part 1：硬性规则

### 规则 1：性能建议必须量化

定义
每条优化建议必须包含预期的性能收益描述（如"减少每帧 N 次分配"、"降低查找复杂度从 O(n) 到 O(1)"），不能只说"这样更快"。

检查清单
- [ ] 每条建议是否有量化或定性的收益描述？
- [ ] 是否区分了"必须优化"和"建议优化"？

### 规则 2：优化不得改变功能语义

定义
性能优化建议不能改变代码的功能行为。如果优化方案会影响功能，必须明确标注并让用户决策。

检查清单
- [ ] 优化后的代码功能是否与原来一致？
- [ ] 有副作用的优化是否标注了风险？

### 规则 3：平台参数必须匹配

定义
性能红线和优化策略必须与 L1 参数（target_platform, optimization_goal）匹配。禁止用 PC 标准审查 Mobile 代码。

检查清单
- [ ] 是否按 target_platform 选择了正确的性能红线？
- [ ] 优化建议是否符合 optimization_goal？

## Part 2：常见陷阱

### 陷阱 1：过度优化低频代码

现象
花大量精力优化初始化代码或低频调用路径，而忽略了真正的热路径。

防护
先确认代码的调用频率（每帧？每秒？仅初始化？），优先优化高频路径。

### 陷阱 2：优化引入新 bug

现象
为了性能将同步改异步，或加入缓存但未处理失效，导致功能异常。

防护
每条优化建议必须评估是否引入新的边界条件。复杂优化建议配合 debug-skill 检查。
