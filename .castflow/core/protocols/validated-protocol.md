# Validated Protocol - 用户接受/拒绝信号

> **加载时点**：仅 T3-FEEDBACK（用户给出明确反馈时）。完整时点定义见项目根 `CLAUDE.md`「使用Skill的分层加载」段。
> **关联**：进化系统通过 validated 信号识别有效经验和反面教材。

---

## 协议定义

AI 判断用户明确接受或拒绝操作结果时，写入 `traces/.pending_validated.json`。trace-flush 读取后注入 trace 条目的 `validated` 字段。

## 写入格式

```json
{"result": "accepted"}
```

或：

```json
{"result": "rejected"}
```

| `result` | trace `validated` |
|----------|-------------------|
| `accepted` | `true` |
| `rejected` | `false` |

## 触发判断

**接受信号（写 accepted）**：
- 用户明确确认（"好"、"OK"、"可以"、"对的"、"就这样"）
- 用户开始使用结果（"那我去测了"、"提交了"、"合并了"）
- 用户验证后表达满意（"测过了没问题"、"运行正常"）

**拒绝信号（写 rejected）**：
- 用户要求重做或修改（"重写"、"改一下"、"不对"、"再来"）
- 用户指出错误（"这里报错"、"没考虑 X"、"漏了 Y"）
- 用户明确否定（"不行"、"不对劲"、"撤销"）

## 严格的"不确定不写"原则

**宁可漏报，不要误报**。模糊反馈不触发：

| 用户反馈 | 是否触发 |
|---------|---------|
| "好" | accepted |
| "好的" | accepted |
| "可以" | accepted |
| "还不错" | **不写**（模糊） |
| "先这样" | **不写**（暂时妥协） |
| "嗯" | **不写**（无意义回应） |
| "我看看" | **不写**（未表态） |
| "重做" | rejected |
| "改一下 X" | rejected |
| "再加点 Y" | **不写**（增量需求，非否定） |

## 文件位置

`<项目根>/.claude/traces/.pending_validated.json`（覆盖写）
