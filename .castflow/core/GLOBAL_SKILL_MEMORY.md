# GLOBAL_SKILL_MEMORY - Skill 运行时核心协议

> **性质**：所有 Skill 执行的底层法律。运行时约束，非创建规范。
> **关联文档**：创建和迭代规范见 [SKILL_ITERATION.md](./SKILL_ITERATION.md)
> **扩展协议**：使用 code-pipeline 时，额外加载 [pipeline_protocol.md](./code-pipeline-skill/config/pipeline_protocol.md)

---

## 文档定位说明

本文档定义 Skill **执行时（运行时）** 的核心约束协议，适用于所有项目、所有 Skill。

**包含**：
- 协议 1：API 物理验证（防止幻觉 API）
- 协议 2：学习后约束对齐（防止约束遗漏）

**与其他文档的关系**：
- SKILL_ITERATION.md = Skill 文件**创建和迭代**时遵守的规范（结构、格式、命名）
- GLOBAL_SKILL_MEMORY.md = Skill **执行时**遵守的核心约束协议（本文档）
- pipeline_protocol.md = code-pipeline 执行时的扩展协议（仅 pipeline 场景加载）

## 协议 1：物理证据准入制 - P0

**定义**：禁令与 API 验证必须物理可见。本协议是 CLAUDE.md "API验证最高优先级规则(P0)" 的运行时执行细则。

**类/功能使用的强制前置条件**：

任何类或功能的使用必须遵循此三段式验证流程，缺一不可：

1. **阅读逻辑** - 理解该类的作用
   - 打开该类的源代码文件（Read）
   - 理解该类的职责、成员变量、生命周期
   - 理解该类与其他类的关系
   - 如果是自定义类（非标准库），必须完整阅读，不能只看方法签名

2. **掌握API** - 知道该类的真实API
   - 在打开的源代码中识别所有public方法和属性
   - 了解每个API的参数类型、返回值、副作用
   - 特别注意：自定义类往往有特殊的API模式（如AddListener而非ClickEvent）
   - **禁止**：假设"这个类应该有某个方法"或"其他类有的方法这个类也应该有"

3. **验证存在** - 确认该API真的存在于这个类
   - 在已打开的源文件中找到该API的定义（不是别的类）
   - 用Grep验证该API的至少一个真实使用案例
   - 记录API来源：该方法在第几行、返回类型是什么、需要什么参数

**常见违规的本质**：
- **跨类污染**：看到一个类有某个API，就假设同类型的其他类也有
- **记忆假设**：基于"类似组件通常的做法"来编写代码，而不是查证这个特定类
- **部分验证**：看到grep结果有相关字符串就认为API存在，没有打开源文件确认是该类定义的
- **类型假设**：假设字段类型应该是什么，而不是查看项目现有代码的真实做法

**必须做法**：
- 使用任何API前，必须能说出"这个API在文件X的Y行定义"
- 每个API调用在代码中应有源码位置的注释
- 如果无法完成三段式验证，不生成代码，立即停止并标记TODO

## 协议 2：学习后强制约束对齐 (Before Code Generation) - P0

**定义**：当完成任何代码学习、API提取或实现方案设计后，在生成/修改代码前，必须执行约束对齐检查。

**适用Skill**：所有生成代码的Skill（programmer-*、debug-skill、architect-skill、code-pipeline等）

**背景问题**：
在学习现有项目代码时，注意力容易被吸引到"怎么用了这个API"，导致生成代码时遗漏SKILL_MEMORY或CLAUDE.md中的约束（变量命名、组件限制、API使用规范等）。本协议通过强制"学习->对齐->生成"的三阶段流程解决该问题。

**执行流程**（必须按顺序）：

**第1步：列出所有适用的约束来源**
- [ ] GLOBAL_SKILL_MEMORY.md（通用协议）
- [ ] 目标Skill的SKILL_MEMORY.md（该Skill的硬性规则、禁止事项）
- [ ] CLAUDE.md（项目全局规范）
- [ ] 创建清单，列出所有适用约束（如："命名规范使用_camelCase"、"UI禁止使用Image"）

**第2步：对比学习到的实现与约束**
- [ ] 逐项检查学习到的代码是否违反已列出的约束
- [ ] 标记所有冲突，记录"违反项->约束来源"的映射
- [ ] 如有多个冲突，逐个评估是否可以调整实现方式

**第3步：冲突处理与调整**
- [ ] 有冲突 -> 调整实现方式使其符合约束（例如：学习代码用mLevel，改为_level）
- [ ] 无冲突 -> 记录最终的实现方式和应用的约束
- [ ] 有undefined API -> 按CLAUDE.md的规则处理（TODO标记）

**第4步：代码生成前通知用户**
- [ ] 明确列出所有应用的约束
- [ ] 说明任何实现调整的理由
- [ ] 获得用户确认后再生成代码

**检查清单**（代码生成前后都必须）：

- [ ] 是否完成了第1步的约束来源清单？
- [ ] 是否完成了第2步的冲突对比？
- [ ] 是否处理了第3步的所有冲突？
- [ ] 是否在第4步告知用户约束和调整？
- [ ] 生成的代码是否仍违反任何约束？
- [ ] 是否有任何未验证的API（应该用TODO标记）？

**常见错误**：

1. 跳过约束对齐：学习完代码直接生成，导致生成代码违反SKILL_MEMORY规范
2. 部分对齐：只检查了命名规范，忽视了组件限制或API弃用
3. 默认沿用项目旧代码的做法：学习代码本身就违反规范，但直接沿用
4. 未向用户说明调整：做了调整但没有明确告知用户原因和内容
