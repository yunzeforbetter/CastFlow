=====================================
【Skill 创作元规范 _AUTHORING_GUIDE】
=====================================

本文件是 `templates/skills/*.template/README.md` 的**共享上游**。
四份领域 README（architect / debug / profiler / programmer）只描述**领域差异**，
其余通用约束、产出门槛、文件分配、反风格、Rubric 全部以本文件为准。

适用对象：
  • skill-creator 在生成新 Skill 时
  • 任何对既有 Skill 做大规模重写时

阅读顺序（强烈建议）：
  §0 → §2.5 → §8 → §6.b → §9 → §A（Rubric）

━━━━━━━━━━━━━━━━━━━━━━━━━━━
§0 Meta 角色（写给 skill-creator，不是写给最终 Skill）
━━━━━━━━━━━━━━━━━━━━━━━━━━━

你不是在做该领域的工作，你是在为该领域写一份**可被未来反复调用**的 Skill。

身份：
  • Skill 工程师 + 项目知识考古学家
  • 输出物：一套四件套（SKILL / EXAMPLES / SKILL_MEMORY / ITERATION_GUIDE）
  • 验收人：未来的 Agent 与 Pipeline，不是当下的对话方

核心要求：
  • 不写"我能做的事"，只写"项目里真实存在、未来可被检索的事实"
  • 不写"通用最佳实践"，只写"在本项目代码与规则约束下成立的实践"
  • 输出物必须能被 grep / 路由 / DoD 校验，否则视为不合格

━━━━━━━━━━━━━━━━━━━━━━━━━━━
§2.5 项目勘察（Project Recon）— 在动笔前强制执行
━━━━━━━━━━━━━━━━━━━━━━━━━━━

「无勘察不出稿」。在写任何一个字之前，至少完成以下读取：

必读源（缺一不可）：
  □ 仓库根 README / CLAUDE.md（项目宪法）
  □ .cursor/rules/*.mdc（项目硬规则）
      重点：UNITY_GUIDELINES.mdc / UNIFIED_WORKFLOW.mdc / AI_ASSISTANT_GUIDE.mdc
  □ 与本 Skill 同主题的源代码目录
      programmer-<模块>：Assets/Scripts/<模块>/ 全量浏览
      architect       ：所有 Manager/Service 的命名空间分布
      profiler        ：项目性能预算（平台、目标帧率、包体）
      debug           ：项目典型错误模型与日志规范
  □ 至少一个"金标 Skill"作为风格参照
      推荐：.claude/skills/programmer-battle-skill/
            .claude/skills/programmer-eventflow-skill/
  □ 同目录的 *.template.md 四件套结构（不可破坏其槽位）

抽取产物（内部，不输出但必须形成）：
  Recon Notes
    F-001 [事实] 路径/类/规则 → 一句话说明
    F-002 [事实] ...
    A-001 [假设] 当...时（待确认）
  
后续每条结论必须回填依据：
  形如：见 F-003 / 见 A-002（标注假设）
  无依据 → **降级或删除**，不允许"看起来合理就写上"

━━━━━━━━━━━━━━━━━━━━━━━━━━━
§2.b 反触发与让位（Routing Hygiene）
━━━━━━━━━━━━━━━━━━━━━━━━━━━

每份 Skill 的 §2 必须包含两类列表：

  正触发（已有）：本 Skill 应该被召出的关键词与场景
  反触发（必加）：看起来像但不归我管的关键词与场景

让位规则（cross-skill boundaries）：
  涉及"具体性能数据/帧时间/分配"     → 让位 profiler-skill
  涉及"具体模块 API 用法"             → 让位 programmer-<模块>-skill
  涉及"找出某段代码错在哪里"         → 让位 debug-skill
  涉及"该不该这么设计/放在哪一层"    → 让位 architect-skill

写法约束：
  • 反触发条目数 ≥ 正触发的 30%
  • 让位时必须点名对方 Skill 的 id

━━━━━━━━━━━━━━━━━━━━━━━━━━━
§4.x 每个 STEP 的"不足兜底"
━━━━━━━━━━━━━━━━━━━━━━━━━━━

领域 README 的 §4 流程中，每个 STEP 都必须显式声明：

  当输入/证据不足时：
    → 不允许硬上结论
    → 走以下兜底之一：
        a) 退到上一 STEP，先给"测量计划"或"勘察清单"
        b) 把结论降级为"假设 + 待确认"
        c) 主动追问最小必要信息（≤3 个问题）

形式范例（写到 §4 各 STEP 末尾）：
  └─ 不足兜底：若 ___ 不可得 → 切换到 ___，禁止 ___

━━━━━━━━━━━━━━━━━━━━━━━━━━━
§6.b 交付门槛（Definition of Done）
━━━━━━━━━━━━━━━━━━━━━━━━━━━

skill-creator 提交前必须自检以下硬门槛，任一不满足即重写：

通用 DoD：
  D1  SKILL.md ≤ 200 行（导航文件，不堆细节）
  D2  EXAMPLES.md 至少 3 个**项目真实**类名/路径，且当前可被 grep 命中
  D3  SKILL_MEMORY.md 至少 5 条形如「必须 / 禁止 / 易踩」的可执行规则
  D4  ITERATION_GUIDE.md 写明「何时新增 / 何时合并 / 何时退役」
  D5  正/反触发词比 ≤ 3:1（即反触发足够）
  D6  每条结论可追溯到一条 Recon 事实编号
  D7  零虚构：所有引用的类/路径/规则文件必须真实存在
  D8  四件套之间无内容重复（同一事实只在一处出现）

领域附加 DoD（在各领域 README 末尾追加）：
  architect：必须给出"架构选项树"和"不采纳的后果"
  debug    ：每个 P0/P1 必须配最小可运行复现 case
  profiler ：必须绑定数字化预算（如 0.5ms / 0 alloc / FPS 目标）
  programmer：必须声明"我对哪些路径负责" + "与邻模块的 API 边界"

拒绝交付（自动 reject）：
  R1  发现编造类名/路径
  R2  发现复制粘贴官方教科书段落（无项目化改写）
  R3  正文出现 emoji / 装饰性符号（与本仓库风格不符）
  R4  SKILL.md 内出现「待补充」「TODO」「示例占位」未替换

━━━━━━━━━━━━━━━━━━━━━━━━━━━
§8 文件分配契约（Artifacts Map）
━━━━━━━━━━━━━━━━━━━━━━━━━━━

同一 Skill 的内容必须按下表归位，**单一事实只能出现在一处**：

  内容类型                       归属文件
  ─────────────────────────────  ───────────────────
  导航 / 定位 / 触发 / 反触发    SKILL.md
  对外口径 / 与其他 Skill 的边界 SKILL.md
  ─────────────────────────────  ───────────────────
  强制规则 / 命名约束 / 易错坑   SKILL_MEMORY.md
  数字化预算 / 红线 / 不变量     SKILL_MEMORY.md
  ─────────────────────────────  ───────────────────
  完整可运行示例（用项目真实类） EXAMPLES.md
  典型实现模式与对比代码         EXAMPLES.md
  ─────────────────────────────  ───────────────────
  何时新增 / 退役 / 合并 Skill   ITERATION_GUIDE.md
  与邻 Skill 的演进协调          ITERATION_GUIDE.md

漂移防御：
  • 同一规则在 SKILL.md 与 SKILL_MEMORY.md 同时出现 → 删 SKILL.md 那份
  • 示例代码出现在 SKILL.md 内联 > 5 行 → 移到 EXAMPLES.md
  • 触发词列在 SKILL_MEMORY → 移回 SKILL.md

━━━━━━━━━━━━━━━━━━━━━━━━━━━
§9 反风格清单（Anti-Patterns）— 看到即重写
━━━━━━━━━━━━━━━━━━━━━━━━━━━

A1  "概念科普型"：复述 GoF / DDD / SOLID 等术语而不映射到本项目
A2  "占位符冒充内容"：保留 ___ / TODO / [示例] 等未填项
A3  "全能型角色"：写"我能做架构、性能、调试..."—— 必须收敛到单一职责
A4  "无证据声称"：写「项目使用了 XXX 模式」却给不出代码引用
A5  "通用最佳实践堆砌"：与项目栈/平台无关的泛化建议
A6  "emoji 装饰"：本仓库 README 已统一去除，禁止再引入
A7  "巨型 SKILL.md"：把 EXAMPLES 的内容塞进 SKILL.md
A8  "孤岛 Skill"：未声明与其他 Skill 的让位关系

━━━━━━━━━━━━━━━━━━━━━━━━━━━
§A 评审 Rubric（10 项 × 1～5 分，总分 ≥ 42 才算交付）
━━━━━━━━━━━━━━━━━━━━━━━━━━━

skill-creator 提交前必须按下表自评，并把分数写入提交说明：

  R01 项目贴合度       — 引用了多少项目真实事实（路径、类、规则）
  R02 路由健壮性       — 触发/反触发设计是否覆盖典型误判
  R03 单一职责         — 是否清晰避免与邻 Skill 抢活
  R04 可执行性         — 规则是否可被 Agent 机械执行
  R05 可验收性         — 是否有 DoD、测试用例、最小复现
  R06 文件分配纪律     — 四件套槽位是否各司其职
  R07 数字化程度       — 是否给出可量化阈值（行数、ms、alloc 数）
  R08 演进设计         — ITERATION_GUIDE 是否真的可用
  R09 反风格规避       — §9 中各项是否全部规避
  R10 风格一致性       — 与同目录其他 Skill 的措辞/版式一致

评分示意：
  5 分：明显超出基线，可作为后续金标
  4 分：达标且无明显瑕疵
  3 分：可用但有改进空间
  ≤2 分：必须重写该项

总分 < 42 → 不允许提交，回到 §2.5 重新勘察。

━━━━━━━━━━━━━━━━━━━━━━━━━━━
§B 与本仓库既有结构的契约
━━━━━━━━━━━━━━━━━━━━━━━━━━━

四件套槽位（不允许新增/删除文件名）：
  SKILL.md             ← 导航与对外口径
  EXAMPLES.md          ← 项目真实示例
  SKILL_MEMORY.md      ← 强制规则与易错坑
  ITERATION_GUIDE.md   ← 演进与维护

模板来源：
  templates/skills/<role>.template/*.template.md
  生成时直接复制为目标 Skill 同名 .md，再按本指南填充。

与 CastFlow 冷启动的关系：
  不依赖 `bootstrap-assets/prompts/` 下的独立 prompt 文件（已移除）。
  生成/迁移时以 harness 的 `skill-templates/<域>.template/*.template.md` 与**同目录 README** 为域入口，
  以本指南 + `SKILL_ITERATION.md` 为全 skill 元规范；日常迭代、评测、描述优化可配合 `.claude/skills/skill-creator/` 工具流。
